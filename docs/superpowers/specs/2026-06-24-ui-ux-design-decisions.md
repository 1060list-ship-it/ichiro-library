## プレイリストの設計思想（重要）

プレイリストは「お気に入り動画の保存」ではなく**キュレーション行為**である。

例：新規ファンが「ハマダ」という人物を知りたい場合、複数の配信から
「ハマダと一郎の出会い」「初めてケンカしたエピソード」「現在の立場が決まった回」などを
選び取り、時系列で並べたナラティブ体験を提供する。

- **ブックマーク** = 「このシーンをプレイリスト候補にする」という中間ストック
- YouTubeのお気に入りとは役割が異なる。お気に入りはYouTube本体でよい。
- ブックマーク単体は目的でなく手段。UIはプレイリスト編集への導線として設計する。
- プレイリストは**公開コンテンツ**。`/playlist/[id]` は認証不要で閲覧可能。編集操作はeditor/adminのみ表示。
- ★ブックマークボタンはeditor/admin権限を持つユーザーのみに表示する（一般ファンには不要）。

### プレイリスト description フィールド（実装必須）

`playlists.description TEXT` カラムはDB設計書に存在する（2026-06-21設計書 line 148）。
このフィールドを**UIで必ず表示・編集できるようにすること**。

理由：タイトルだけでは「なぜこのプレイリストが存在するか」がファンに伝わらない。
プレイリストはキュレーターの意図を持つ編集コンテンツであり、文脈説明文はその核心。

**表示場所**：

- `/playlist/[id]` のヘッダー部分（タイトルの直下）
- トップページのプレイリストカード（hover時またはカード下部に短縮表示）

**編集場所**：

- `/member` のプレイリスト新規作成フォーム（titleと並列のtextareaフィールド）
- `/playlist/[id]` のeditor向けインライン編集（SaveStatusBar連動）

## 確定した設計判断

### A. `/member` ファーストビュー

**決定**: プレイリスト一覧をメインに置く

- 理由: メインタスクは「プレイリストの作成・編集」。ブックマークはその補助。
- ブックマーク一覧はタブ2に配置（またはプレイリスト編集画面内の「ストックから追加」UIに統合可）

---

### B. ★ブックマークボタンの実装

**決定**: `StreamCard` を `<div>` ベースに変更する（batou案）

- 現状: `StreamCard` は `<a>` タグ全体がリンク → `<button>` をネストできない
- 変更内容:
  - `StreamCard` のルート要素を `<a>` → `<div>` に変更
  - 動画タイトル部分のみ `<a>` タグで囲む
  - ★ボタンを `<div>` 内に通常フローで配置
- 理由: 絶対配置（設計書の `BookmarkableStreamCard` 案）はz-indexとモバイルタップ精度の問題が出やすい。HTML的にも `<a>` の中に `<button>` はネストできない。
- **影響範囲**: `StreamCard` を使用する全画面（トップ・検索結果・magazine等）。paz は既存スナップショットテストがあれば先に確認すること。

---

### C. 管理画面レイアウト

**決定**: 4タブ構成に変更する（cc案）

```
[ダッシュボード] [動画管理] [パイプライン] [エンティティ申請]
```

| タブ | 含むコンテンツ |
|------|--------------|
| ダッシュボード | StatCard 3枚 + **未レビュー動画一覧**（最上位に引き上げ）+ 字幕取得失敗一覧 |
| 動画管理 | 検索フォーム + 全動画一覧（確認済み含む） |
| パイプライン | ジョブ登録ボタン群 + JobTable |
| エンティティ申請 | `entity_word_requests` 一覧・承認UI（今回実装） |

- 既存の `AdminPageClient.tsx` 内のタブ切り替えボタンパターンを汎用化して `TabNav` コンポーネントを作る
- アクティブタブ: アンダーライン式（`border-b`）。既存の反転スタイル（`bg-white text-gray-950`）を流用

---

### D. カテゴリ × 年フィルター

**決定**: 両立する（排他制御なし）

- 理由: 怪獣の歌唱配信は桁違いの再生数でランキング上位を独占する。年を絞ることで実態に即したランキングが見える。カテゴリ × 年の組み合わせはユーザーにとって有意義。
- 現状の実装をそのまま維持する。

---

### E. エンティティ申請タブ（`/member` 内）

**決定**: 今回実装する

- `/member` のタブ2として `[エンティティ申請]` を配置
- 申請フォーム（エンティティ名・読み・種別・根拠URL）と自分の申請履歴一覧を表示
- 申請承認は管理画面（エンティティ申請タブ）で行う
- 一幾のみ利用の現状でも、将来editor追加時のインフラとして今回実装する

---

### F. 再生済みマーク通知（`/playlist/[id]`）

**決定**: 実装者（paz）判断に委ねる

- localStorage（ブラウザキャッシュ）で保持するため、別端末・シークレットモードでは消える
- 「このデバイスのブラウザのみで保持されます」の文言をどこかに出せばよい
- 初回トースト / ページ下部常時注記 どちらでも可。UIの流れで自然な方を選ぶこと

## cc追加UI提案（Codex検証対象）

以下は今回のセッションでccが追加提案したUI改善。設計書には含まれていなかった内容。
実装前に Codex（paz）が既存コードと照合して採否を確定すること。

### StreamCard 情報積み順の変更

**現状**: 日付 → タイトル → 要約 → タグ  
**提案**: タイトル → (日付 + チャプター数) → タグ → 要約

- タイトルを最上位に。日付は文脈補足に過ぎない
- タグの方が視認性が高くカテゴリ確認が早い
- 再生数・コメント数はランキング表示時以外は省略可

### ReviewBadge / TranscriptBadge（管理画面）

管理画面の `StreamRow` に状態バッジを追加。既存 `JobStatusBadge` と同じパターンで実装。

| バッジ | 状態 | 色 |
|--------|------|-----|
| TranscriptBadge | 取得済み | emerald |
| TranscriptBadge | 未取得 | gray |
| TranscriptBadge | 取得失敗 | red |
| ReviewBadge | 確認済み | emerald |
| ReviewBadge | 未レビュー | yellow |

## `/playlist/[id]` レイアウト仕様（cc案）

```
PC（grid-cols-[55%_1fr]）:
┌──────────────────┬───────────────────────┐
│ YouTubeプレイヤー │ プレイリスト動画リスト  │
│  aspect-video    │  overflow-y: auto      │
│  sticky top-0   │  height: 100dvh-header │
└──────────────────┴───────────────────────┘

スマホ（1カラム）:
プレイヤー（上）→ 動画リスト（下）
```

- iOS Safariの `position: sticky` 問題は右カラムを `overflow-y: auto` + `height: calc(100dvh - header)` で制御
- `PlaylistStreamRow` 構成: `[番号] [サムネイル 80px] タイトル 再生数・日付`
- `StreamCard` は流用しない（設計書と同じ方針）

## トップページ プレイリストセクション仕様（cc案）

```
[カテゴリナビ]
[SearchBar]
[年フィルタ]
──────────────────── ← ここに追加
マイプレイリスト
  [サムネイル] [サムネイル] [サムネイル]   → もっと見る
────────────────────
最近の配信 / カテゴリ結果
[StreamCard] ...
```

- `view === 'top'` かつ `!isSearching` のときのみ表示
- `grid grid-cols-3` 固定・最大3件 + もっと見るリンク（`/member`へ）
- サムネイル: `playlist_streams` の先頭動画の `thumbnail_url`
- 未ログイン時は非表示（`session` が null の場合）

## 並び替え保存フィードバック仕様（batou案）

設計書の「全行disabled + スティッキーヘッダー」に以下を追加:

- 操作した行の右端に `保存中…` インライン表示（保存完了後に消える）
- `SaveStatusBar`（スティッキー）の表示と組み合わせて「動いている」感を担保
- 保存失敗時は操作行とヘッダー両方にエラー表示
